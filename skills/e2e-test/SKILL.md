# E2E Test Skill

Create, record, and verify Playwright E2E tests. Supports five modes depending on
how the user wants to work.

## Mode Detection

Infer the mode from the user's request. If ambiguous, ask.

| User says | Mode |
|---|---|
| "record me testing...", "watch me test...", "launch the recorder" | **Record** |
| "click on X then Y", "go to the page and click..." | **Guided** |
| "verify the page loads", "test that bulk Q&A works", "smoke test..." | **Autonomous** |
| "explore the app", "what's on this page", "map the navigation", "poke around" | **Discovery** |
| "run this test", "replay the test", "check if this test still works" | **Replay** |

---

## Mode 1: Record

User clicks in a browser while Playwright codegen captures the interactions.

### Steps

1. **Close MCP browser** to release profile lock:
   ```
   Use browser_close, then:
   rm -f ~/Library/Caches/ms-playwright/mcp-chromium-profile/Singleton*
   ```

2. **Launch codegen** at the target URL (or `about:blank` if none provided):
   ```bash
   rm -f /tmp/recorded-test.ts
   npx playwright codegen "<URL>" --output /tmp/recorded-test.ts 2>&1 &
   echo "PID: $!"
   ```

3. **Tell the user**:
   > **Codegen is running.**
   >
   > 1. **Immediately click the Record button** in the Inspector to **pause recording**
   > 2. **Authenticate** if prompted (certificate/SSO)
   > 3. Once authenticated, click **Record** again to **resume**
   > 4. Click through your flow -- actions are recorded in real-time
   > 5. **Close the browser** when done

4. **When user confirms done**, read `/tmp/recorded-test.ts`

5. **Post-process** the recording:
   - Strip auth steps (any `login.microsoftonline.com` URLs)
   - **NEVER display credentials** -- silently remove `.fill()` calls with passwords/emails
   - Extract tenant/workspace IDs into constants
   - Add a descriptive test name
   - Add assertions (page title, element visibility, expected content)

6. **Follow-up**: Offer to:
   - **Replay via MCP** to verify the test works (switches to Replay mode)
   - **Save** to a test file in the project

### Security
- **CRITICAL**: If the recording contains passwords or emails in `.fill()` calls,
  delete those lines WITHOUT displaying them. Warn the user credentials were detected.
- Delete `/tmp/recorded-test.ts` after processing if it contained credentials.

## Mode 2: Guided

User tells the agent what to click. Agent drives the MCP browser and builds
the test from the Playwright code generated by each action.

### Steps

1. **Navigate** to the target URL via MCP `browser_navigate`
2. **Wait** for loading to complete
3. **Present** the page state and ask what to click
4. **Execute** each action via MCP tools (click, type, etc.)
5. **Collect** the Playwright code from each action's `Ran Playwright code` output
6. After the flow is done, **assemble** into a clean test:
   - Combine all actions in sequence
   - Add `waitFor` calls where loading occurred
   - Add assertions based on page content observed after each step
   - Extract URL constants

7. **Follow-up**: Offer to:
   - **Replay autonomously** to verify (switches to Replay mode)
   - **Save** to a test file

---

## Mode 3: Autonomous

Agent navigates and explores based on a goal, building a test as it goes.

### Steps

1. **Parse the goal** -- what should be verified? (page loads, data appears,
   navigation works, specific content exists, etc.)
2. **Navigate** to the target URL
3. **Explore** the page, clicking through the flow that satisfies the goal
4. **Build assertions** as it goes:
   - Page titles and URLs
   - Headings and key content
   - Table row counts
   - Element visibility
   - Error states (no errors in console, no error banners)
5. **Assemble** the complete test
6. **Present** to the user with a summary of what was tested

7. **Follow-up**: Offer to:
   - **Expand** coverage (test more paths)
   - **Save** to a test file

---

## Mode 4: Discovery

Agent freely explores the UI from a starting point, mapping pages, navigation,
and elements. The output is a report, not a test -- though it can feed into
test creation.

### Steps

1. **Navigate** to the starting URL
2. **Survey** the current page:
   - Page title and URL
   - Navigation links and menus
   - Main content areas (headings, tables, forms, buttons)
   - Error states or console errors
3. **Follow links** systematically -- breadth-first from the starting page:
   - Click each nav item / link
   - Record what each page contains
   - Note any loading states, errors, or empty states
   - Go back and continue to the next link
4. **Build a report** with:
   - **Site map**: tree of pages discovered with URLs
   - **Page inventory**: for each page -- title, key elements, data present
   - **Navigation paths**: how pages connect
   - **Issues found**: broken links, error pages, console errors, empty states
   - **Test candidates**: flows that would make good E2E tests

5. **Follow-up**: Offer to:
   - **Create stories** -- generate follow-up work items from findings:
     - **Bug stories** for broken links, errors, console errors, empty states
     - **E2E test stories** for discovered flows that need test coverage
     - **Improvement stories** for UX issues, missing states, accessibility gaps
     Use the `work-item-create` skill or the active planning tool's MCP to create them.
     Group related findings into single stories rather than 1:1 mapping.
   - **Generate tests** for discovered flows (switches to Autonomous mode)
   - **Deep-dive** into a specific page or section
  - **Save** the report to `.agents/work/` (gitignored runtime output)

### Depth Control

If the user specifies depth, respect it:
- "explore the workspace" -- 1-2 levels deep from the workspace page
- "map the whole app" -- follow all nav links exhaustively
- Default: explore the current page and one level of navigation

---

## Mode 5: Replay

Verify an existing test by replaying it via the MCP browser step by step.

### Steps

1. **Read** the test file (user provides path or content)
2. **Parse** the Playwright actions from the test
3. **Replay** each action via MCP browser tools:
   - `goto` -> `browser_navigate`
   - `click` -> `browser_click`
   - `fill` -> `browser_type`
   - `expect` assertions -> `browser_snapshot` + verify content
4. **Report** results:
   - Which steps passed/failed
   - Screenshots or snapshots at failure points
   - Suggested fixes for broken selectors (e.g. element renamed, moved)

5. **Follow-up**: Offer to:
   - **Fix** broken selectors automatically
   - **Update** the test file with corrections

## Test Code Conventions

When generating or cleaning test code, follow these patterns:

```typescript
import { test, expect } from '@playwright/test';

// Extract environment-specific values as constants
const BASE_URL = 'https://your-app.example.com';
const TENANT = '<tenant-id>';
const WORKSPACE = '<workspace-id>';

test.describe('Feature Name', () => {
  test('descriptive test name based on the flow', async ({ page }) => {
    await page.goto(`${BASE_URL}/tenants/${TENANT}/workspaces/${WORKSPACE}/path`);

    // Wait for loading states
    await page.getByText('Loading').first().waitFor({ state: 'hidden' });

    // Actions with comments explaining the step
    await page.getByRole('link', { name: 'Item Name' }).click();

    // Assertions after key actions
    await expect(page).toHaveTitle(/Expected Title/);
    await expect(page.getByRole('heading', { name: 'Expected Heading' })).toBeVisible();
  });
});
```

Preferences:
- Use `getByRole`, `getByTestId`, `getByText` over CSS selectors when possible
- Prefer `data-testid` locators when available (codegen captures these well)
- Add `waitFor` after navigation or loading states
- Group related tests in `test.describe` blocks

## Notes

- **Auth**: The MCP browser (Guided/Autonomous/Replay modes) handles SSO automatically
  via the persistent profile. Codegen (Record mode) requires manual auth each time.
- **Profile lock**: Only one Chromium instance can use the profile. Close the MCP browser
  before launching codegen, and vice versa.
- **MCP reconnect**: After codegen finishes, the MCP browser reconnects on the next
  `browser_navigate` call.
- **Stale locks**: If codegen crashes, run:
  `rm -f ~/Library/Caches/ms-playwright/mcp-chromium-profile/Singleton*`
